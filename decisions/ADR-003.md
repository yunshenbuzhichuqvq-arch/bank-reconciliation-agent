# ADR-003：MVP-1 第二批前端技术栈与组件策略

## Status
Accepted（用户 2026-06-05 拍板）。

## Context
- 第一批后端 8 个 task 已实现完毕、正在 review，API 契约视为**冻结**。
- 设计稿 `overall-frontend-design-style.md` 要求 Claude.ai 系的暖中性、低装饰、内容优先气质；`tasks.md` 原定 Element Plus（默认冷灰企业 SaaS 风），两者存在张力。
- 已确认的后端契约缺口（详见各 spec）：
  1. **无任务列表接口**：只有 `GET /reconcile/{task_id}/status`，没有"列出所有 task"。
  2. **无 CORS**：`main.py` 未挂 `CORSMiddleware`，独立 Vite dev server 无法直连后端。
  3. **`/review/pending` 不含金额/不含 task_id/flow_id**：`PendingReviewItem` 只有 AI 建议与 RAG 来源，无法在前端拼出 PRD §4.4 的"左右流水对账面板"。
  4. **错误响应无机器码**：业务/校验错误统一 HTTP 4xx + `{detail:"<英文串>"}`，`ApiResponse.error_code` 定义了但从不赋值。

## Options
**组件策略**
- A. Element Plus 全量主题化：最省事，但 Table 等重组件难脱离 SaaS 观感。
- B. 完全自定义：最贴设计稿，但要自建 Table/Upload/a11y，工作量最大。
- C.（选定）**混合**：EP 仅承接 Upload / Table / Pagination / Dialog / Form 等重组件并按 token 主题化；Button / Badge / Card / AppShell 自定义。

**后端集成**
- 加 `CORSMiddleware`（动 src/）vs **Vite dev proxy**（不动后端）。

**看板数据**
- 新增 `GET /reconcile/tasks`（动后端）vs **单任务看板**（路由 `/tasks/:taskId`，仅展示当前 task）。

## Decision
1. **组件：混合（Option C）**。EP 重组件主题化 + 自定义轻组件，统一吃一套 CSS 变量设计 token。
2. **后端冻结**：本批不改任何 `src/`。开发期跨域用 `vite.config` 的 `server.proxy` 把 `/api` 代理到后端。
3. **单任务看板**：看板按 `task_id` 维度展示，不做多任务列表。
4. **技术选型假设**：Vue3 + Vite + **TypeScript**；**不引入** Pinia / Tailwind / 图表库 / 额外 UI 库。状态用路由参数 + 局部 state + 轻量 composable。
5. **目录**：所有前端代码落在 `frontend/`，与 Python 后端隔离。

## Consequences
- 正面：不触碰正在 review 的后端；前端在冻结契约内自洽；设计气质可控。
- 代价 / 推迟项（后续后端批次再做，不在本批 spec 内）：
  - **多任务列表**推迟（缺 list 接口）。
  - **生产环境 CORS / 同源部署**推迟（MVP-1 仅 dev proxy）。
  - **复核页左右流水对账面板降级**：MVP-1 复核页以"AI 建议 + RAG 来源 + 处置动作"为中心，不展示 Source A/B 金额对比（`/review/pending` 不提供）。这是对 PRD §4.4 的已知偏离，记录在 TASK-204。
- 错误处理：前端按 HTTP status + `detail` 文案兜底，不依赖 `error_code`。
