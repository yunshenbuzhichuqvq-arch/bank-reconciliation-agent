# ADR-004：MVP schema 演进与本地库迁移机制

## Status
Accepted（用户 2026-06-05 拍板，选 Option C）。由第二批前端 smoke 失败触发。

## Context
- **触发事件**：第二批前端端到端 smoke 失败。上传成功（HTTP 200）后，任务看板 `GET /reconcile/{id}/status` 与 `/exceptions` 全部 500。
- **根因（已用 traceback 定位）**：
  ```
  pymysql OperationalError (1054, "Unknown column 't_reconciliation_task.scenario_type' in 'field list'")
  ```
  `scenario_type` / `batch_id` 是 TASK-101 给模型与 `schema.sql` 加的列，但**线上 dev 库 `AI_agent` 从未应用**。`upload` 的 INSERT 只写它赋值的列（不含新列）所以 200；`status`/`exceptions` 走全字段 ORM 查询，SELECT 列表里有 `scenario_type` → 1054 → 500。前端只是如实展示后端 500。
- **为什么会漂移**：项目只有两条建表路径，**都是"建新表、从不改旧表"**：
  1. `src/.../db/schema.sql`：`CREATE TABLE IF NOT EXISTS`
  2. 各 service 运行时 `metadata.create_all(engine, tables=[...])`（同样 IF NOT EXISTS 语义）

  加列后，这两条路对**已存在**的表都是 no-op。后端测试在全新空库上 `create_all`，所以 CI 永远看不到漂移；README 的 `mysql ... < schema.sql` 对老库也是 no-op。
- **附带问题**：schema 被定义两遍（`schema.sql` + 各 service 的 `Table`），AGENTS.md 第 7 条要求手工同步——重复源，本身就是漂移温床。
- **现阶段约束**：MVP-1 本地单人 demo，库内是一次性 mock 数据，可丢弃。

## Options
- **A. Alembic（正规迁移工具）**：autogenerate、版本化、可回滚、保留数据。代价：引依赖；autogenerate 需把分散的 `Table` 收敛到单一 `metadata` 作为 target；schema 仍在频繁变动期，写/审迁移文件有摩擦。**长期正解，但对"数据可丢弃的本地 demo"偏重。**
- **B. 手写幂等 ALTER 迁移文件 + 迷你 runner**：`migrations/00x.sql` + `schema_migrations` 版本表，runner 顺序应用未执行项。比 A 轻、保留数据。代价：MySQL 不支持 `ADD COLUMN IF NOT EXISTS`，幂等要靠 information_schema 判断或容错；要自建 runner 与版本表；手写易漏列。
- **C.（推荐，MVP-1）dev 一键重建脚本 `scripts/reset_db.py`**：drop 所有表 → 用**运行时模型定义**（`metadata.create_all`）重建。等于把这次手动修复固化成脚本。代价：**销毁数据**（dev/demo 可接受；有真实数据的环境不可用）；不提供"保留数据"的增量迁移。

## Decision
1. **MVP-1 选 C**（已拍板）：新增 `scripts/reset_db.py`，约定"改 schema 后，老库跑一次 reset"。理由：数据可丢弃、单人本地、schema 仍 churn，C 成本最低，且与"schema.sql 手工维护"的现状一致。spec 见 [TASK-109](../specs/TASK-109.md)。
2. **重建源用模型 `Table` 的 `metadata.create_all`，不解析 `schema.sql`**：避免 schema.sql 注释含分号导致的解析坑（本次手动修复踩到过——naive `split(";")` 把注释 `... not implemented in this baseline DDL.` 当成 SQL 喂给了 MySQL），且保证重建结果与运行时代码一致。
3. **毕业条件**：一旦出现需要保留的数据（多人协作 / 持久演示 / 预生产），升级到 **A（Alembic）**，并借机消除 schema 双源（让 `Table` 成为唯一来源，`schema.sql` 由迁移生成或废弃）。

## Consequences
- 正面：一条命令把任意漂移的 dev 库拉回与代码一致；本地与 CI 行为统一；不动任何 `src/` 业务逻辑。
- 代价 / 推迟项：
  - reset **清空数据**；"保留数据的增量迁移"推迟到 A 阶段。
  - **schema 双源**（`schema.sql` vs `Table`）未根治，留待 A 阶段统一。
- 关联更新：本次已手动对 dev 库执行等价 reset（drop 全表 + 按 `schema.sql` 重建），smoke 全链路恢复 200。脚本化是为了让这件事可复现、可自助。
