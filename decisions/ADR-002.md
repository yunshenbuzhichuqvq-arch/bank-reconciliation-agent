# ADR-002: MVP-1 schema 采用外科手术式增量，保留 bank/clear 命名

- Status: Accepted (2026-06-05)
- Deciders: 用户（拍板）, Planner agent（提案）
- Related: system-prd.md §6, src/.../services/*.py, src/.../db/schema.sql, TASK-101

## Context

PRD §6 描述的目标 schema 已演进到 `t_source_a_transaction` / `t_source_b_transaction` 命名，且每张表都带 `user_id` / `scenario_type`。
但现网 MVP-0 代码（`services/*.py` 里的 SQLAlchemy `Table` + `db/schema.sql`）仍是：

- 表名 `t_bank_transaction` / `t_clear_transaction`（bank/clear 语义）。
- 多数表**无** `user_id` / `scenario_type` / `exception_branch`。
- 无迁移框架（Alembic 等）；测试靠 `metadata.create_all` 现建表，生产靠手工维护 `schema.sql`。

MVP-1 需要：多租户列、`exception_branch`、新增 `t_human_review` / `t_agent_execution_log`。

## Options

1. **外科手术式增量**：在现有表上加列 + 新增两表，**保留 bank/clear 命名**，`source_a/b` 重命名推迟。
2. **对齐 PRD §6**：把表重命名为 `t_source_a/b_transaction` 并补全所有字段。

## Decision

采用 **Option 1**。理由：

- Option 2 会触及几乎所有 `services/*.py`、`schemas/*.py`、`scripts/`、`tests/`，是一次大重构，超出 MVP-1「补产品闭环」的目标，且与本阶段「不追求完整工程化」一致。
- MVP-1 的多租户与分支能力不依赖表名；`bank`/`clear` 只是物理槽位名。

落地约定：
- 列以**增量**方式添加，给安全默认值（`user_id` 默认 `demo_user`、`scenario_type` 默认 `BANK_ENTERPRISE`），保证 `create_all` 与既有测试不破。
- `db/schema.sql`（MySQL DDL）与各 service 的 SQLAlchemy `Table` 定义**同步**维护，二者是同一 schema 的两个产物。
- A/B ↔ 槽位映射：Source B(银行流水)=`bank_*`，Source A(企业账簿)=`clear_*`（见 ADR-001）。
- 生产 schema 里 PRD §6 提到的 `t_source_a/b` HASH 分区在 SQLite 测试中不适用，仅作为 `schema.sql` 注释保留，不在 MVP-1 实装分区。

## Consequences

- 现网代码与 PRD §6 之间存在**已知命名差异**，本 ADR 即其登记；统一重命名留到后续阶段（V1 容器化时一并处理）。
- 每加一列都要两处改（service Table def + schema.sql），Code Review 需对照检查。
- 新表 `t_human_review` / `t_agent_execution_log` 从一开始就带 `user_id` / `scenario_type`，无历史包袱。
