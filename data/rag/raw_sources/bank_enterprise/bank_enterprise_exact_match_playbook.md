# 银企对账精确匹配与自动平账规则（演示）

> 本文件为演示规则，非真实银行内部制度。判定依据仅标注公开制度方向，供 RAG 检索演示使用。

## [BE-R001] 精确匹配自动平账主规则

- 适用场景：银行流水（source_b）与企业账簿（source_a）存在同一业务标识，金额、方向、对手与摘要四要素一致，属可直接闭环的正常对账。
- 判定条件：`source_a.amount == source_b.amount`（amount_diff == 0）；交易日期落在同一日切日；flow_id 或对手账号可关联匹配；借贷方向一致；摘要与客户名称一致（narrative_or_name_mismatch == false）。
- 处理建议：命中 AUTO_FIX，直接写入对账台账，不进入 Agent / RAG 链路；保留匹配字段快照供事后抽检。
- 判定依据：参考方向《企业内部控制基本规范》"账实相符、账账相符"原则，以及人民银行支付结算制度（公开制度方向，非内部制度）。
- 示例：企业账簿与银行流水均含流水号 `20240615-008812`、金额 ¥12,500.00、对手"杭州XX科技有限公司"，方向与摘要一致 → 自动平账。

本项目抽象规则：`EXACT_MATCH` 对应 `BE-R001`（动作 AUTO_FIX，无 error_type）。仅当金额、方向、流水/对手、摘要户名四要素全部一致时才允许自动平账。

## [BE-R001] 四要素全一致的自动平账判定细则

- 适用场景：需要逐字段确认"精确"成立，避免把"金额相近"误判为"金额一致"。
- 判定条件：金额按分位精确比对（不做四舍五入容差）；交易方向（借/贷）一致；流水号或银行参考号可一一对应；对手户名规范化后完全一致。任一字段不满足即不得判为精确匹配。
- 处理建议：四要素满足→AUTO_FIX；任一不满足→交由对应异常分支路由，不在本规则内"放宽通过"。
- 判定依据：参考方向会计基础工作规范中原始凭证要素完整、可核对的要求（公开制度方向，非内部制度）。
- 示例：金额、方向、流水号一致但户名"杭州XX科技" vs "杭州XX科技有限公司" → 不命中精确匹配，转 BE-R004 复核。

本项目抽象规则：精确匹配是"零容差"判定，任一要素不符不得自动平账，须显式路由到异常分支。

## [BE-R001] 金额一致但摘要或户名不一致不得自动平账（边界）

- 适用场景：金额、方向、流水均对得上，但摘要语义或客户名称对不上，看似可平实则需语义核对。
- 判定条件：amount_diff == 0 且 flow_matched == true，但 narrative_or_name_mismatch == true。
- 处理建议：不得 AUTO_FIX；路由到摘要/客户名不一致分支，先由 ExtractionAgent 结构化摘要与户名，再交 AuditAgent 判定。
- 判定依据：参考方向银行结算账户管理制度中户名与账户一致性核验要求（公开制度方向，非内部制度）。
- 示例：金额 ¥8,000.00、流水号一致，但摘要"货款"vs"预付款"、户名简称与全称不符 → 转 BE-R004，不自动平账。

本项目抽象规则：金额一致不足以自动平账；摘要或户名不一致须转 `NARRATIVE_NAME_MISMATCH`（BE-R004），与精确匹配（BE-R001）严格区分。

## [BE-R001] 疑似精确实则存在手续费尾差的排除（干扰：vs BE-R003）

- 适用场景：双边金额"几乎一致"，差额恰好等于手续费或税费特征金额，容易被误判为精确匹配。
- 判定条件：amount_diff != 0 且差额落在手续费/税费特征集（如 0.50/1.00/2.00/5.00/10.00/100.00 元）→ 不属于精确匹配。
- 处理建议：不得 AUTO_FIX；按手续费/税费差异分支处理，要求带出手续费字段、净额与全额口径证据。
- 判定依据：参考方向人民银行支付结算手续费管理相关公开规定（公开制度方向，非内部制度）。
- 示例：企业记 ¥10,000.00、银行入账 ¥9,995.00，差额 ¥5.00 命中手续费特征集 → 转 BE-R003，而非精确匹配。

本项目抽象规则：差额命中手续费/税费特征集时归 `FEE_TAX_DIFF`（BE-R003），不得按精确匹配自动平账，二者以"差额是否为零"严格切分。

## [BE-R001] 同日同额但流水或对手无法关联的处理（边界：vs BE-R007）

- 适用场景：金额一致、同一日切日，但缺少可一一对应的流水号或对手账号，无法构成"精确"。
- 判定条件：amount_diff == 0 且同日，但 flow_id 无法一一匹配、对手账号仅部分相似（counterparty_account LIKE counterparty_key）。
- 处理建议：不得 AUTO_FIX；按跨流水候选匹配处理，输出候选并交人工/Agent 确认。
- 判定依据：参考方向会计档案与凭证可追溯要求（公开制度方向，非内部制度）。
- 示例：两笔均为 ¥3,200.00、同日，但银行侧无流水号、对手仅"XX贸易"近似 → 转 BE-R007 候选匹配，不自动平账。

本项目抽象规则：精确匹配要求流水/对手可一一关联；仅金额同日相近而缺乏唯一关联时归 `FUZZY_MATCH_CANDIDATE`（BE-R007）。

## [BE-R001] 多笔合并对一笔的精确匹配边界（边界：vs 部分到账）

- 适用场景：企业一笔总账对应银行多笔到账（或反之），合并后金额一致，需判断能否整体自动平账。
- 判定条件：多笔金额之和 == 单笔金额且对手一致、同一结算周期内；若存在缺口或时间跨期则不成立。
- 处理建议：合并后四要素一致可平账，但须在台账留存"多笔合并"关系；存在缺口→按金额差异或单边残留处理。
- 判定依据：参考方向收付实现与配比核对的会计基础工作规范方向（公开制度方向，非内部制度）。
- 示例：银行三笔 ¥3,000 + ¥3,000 + ¥4,000 合计 ¥10,000，对应企业一笔 ¥10,000、对手一致 → 合并平账并标注关系。

本项目抽象规则：合并匹配在金额闭合且对手一致时可自动平账，但须保留合并关系；出现缺口转 `AMOUNT_MISMATCH`（BE-R002）。

## [BE-R001] 跨币种折算后金额一致的自动平账谨慎性（边界）

- 适用场景：原币与本币间折算后金额一致，但折算口径、汇率与折算日期可能引入隐性差异。
- 判定条件：本币折算额一致且原币、汇率、折算日期均可核对一致时方可视为精确；任一口径缺失即不判精确。
- 处理建议：原币与汇率证据齐全→可平账；缺汇率或折算日期→不自动平账，留待人工或规则确认。
- 判定依据：参考方向外币业务折算与汇率披露的会计核算方向（公开制度方向，非内部制度）。
- 示例：USD 1,000 按 7.20 折算 ¥7,200，企业与银行折算一致且汇率、日期可核对 → 可平账。

本项目抽象规则：跨币种"金额一致"须以原币、汇率、折算日期三项可核对为前提，否则不得按 BE-R001 自动平账。

## [BE-R001] 自动平账的证据留存与可回溯要求（处理规范）

- 适用场景：任何 AUTO_FIX 闭环后的证据归档，确保自动平账可被审计回溯。
- 判定条件：自动平账完成时须固化匹配字段快照（金额、方向、流水/对手、摘要、命中规则 ID 与时间戳）。
- 处理建议：将快照写入对账台账并标注 `AUTO_FIX by BE-R001`；抽检发现要素缺失的应回退为人工复核。
- 判定依据：参考方向会计档案管理与审计留痕要求（公开制度方向，非内部制度）。
- 示例：平账记录附 `{rule_id: BE-R001, amount_diff: 0, matched_fields: [...], ts: ...}` 供抽检。

本项目抽象规则：自动平账不等于"无需留痕"；BE-R001 闭环必须保留可回溯的匹配快照与命中规则标记。
