# 银企对账跨流水候选匹配规则（演示）

> 本文件为演示规则，非真实银行内部制度。判定依据仅标注公开制度方向，供 RAG 检索演示使用。

## [BE-R007] 跨流水候选匹配主规则

- 适用场景：缺少可一一对应的流水号，但金额、日期与对手高度相似，可作为候选匹配交由确认。
- 判定条件：`source_a.amount == source_b.amount` 且 `source_a.trade_time.date() == source_b.trade_time.date()`，且对手账号近似（`counterparty_account LIKE '%' + counterparty_key + '%'`），但无唯一流水号关联。
- 处理建议：产出候选匹配（EXCEPTION，error_type=FUZZY_MATCH_CANDIDATE），优先级介于精确匹配与单边规则之间；交人工或 Agent 确认后才落账。
- 判定依据：参考方向账证、账账核对与可追溯凭证管理方向（公开制度方向，非内部制度）。
- 示例：两笔均为 ¥3,200.00、同日，对手"XX贸易" vs "XX贸易(上海)" → 候选匹配待确认。

本项目抽象规则：`FUZZY_MATCH_CANDIDATE` 对应 `BE-R007`；候选匹配不自动平账，必须经确认环节闭环。

## [BE-R007] 候选匹配的相似度与确认环节

- 适用场景：对候选匹配排序并控制误匹配风险，明确"候选"到"确认"的链路。
- 判定条件：综合金额一致性、日期邻近度与对手相似度计算候选可信度；可信度不足或存在多个等价候选时不得自动确认。
- 处理建议：高可信单一候选→提请确认；多候选或低可信→列出全部候选并转人工，不擅自择一。
- 判定依据：参考方向不相容职务分离与人工复核方向（公开制度方向，非内部制度）。
- 示例：一笔银行流水匹配到两笔等额企业账 → 列两候选转人工，不自动选其一。

本项目抽象规则：候选匹配存在多义时必须转人工择一，禁止系统在等价候选间自动决定。

## [BE-R007] 与精确匹配的区分（干扰：vs BE-R001）

- 适用场景：金额与日期一致时，判断属"可自动平账的精确"还是"需确认的候选"。
- 判定条件：精确=存在唯一流水号/对手一一对应且摘要户名一致；候选=仅相似（LIKE）、缺唯一关联或户名仅近似。
- 处理建议：有唯一关联→走 BE-R001 自动平账；仅相似→走 BE-R007 候选确认，不得按精确自动平账。
- 判定依据：参考方向凭证唯一性与可核对要求（公开制度方向，非内部制度）。
- 示例：金额同日一致但银行侧无流水号、对手仅近似 → BE-R007 而非 BE-R001。

本项目抽象规则：精确匹配与候选匹配以"是否存在唯一一一关联"切分；仅相似关联归 BE-R007，不享受 BE-R001 自动平账。

## [BE-R007] 与重复记账的区分（干扰：vs BE-R008）

- 适用场景：一侧多笔与另一侧一笔金额相近时，判断是"多个候选"还是"疑似重复多记"。
- 判定条件：候选=不同笔业务恰好金额相近、需择一匹配；重复=同一笔业务在一侧被多次入账、净额被双计。
- 处理建议：核对是否为同一业务的多次入账；同一业务多记→BE-R008；不同业务相近→BE-R007 候选。
- 判定依据：参考方向防止重复入账与业务唯一性核对方向（公开制度方向，非内部制度）。
- 示例：企业两笔 ¥5,000 对应两笔不同业务 → 候选择一；若两笔实为同一业务被录两次 → 重复记账。

本项目抽象规则：候选匹配与重复记账以"是否同一业务被多记"切分；同一业务多记=DUPLICATE_BOOKING（BE-R008），不同业务相近=BE-R007。

## [BE-R007] 对手户名近似但账号不同的候选

- 适用场景：对手名称相似（简称/分支机构）但银行账号不同，需谨慎判断是否同一对手。
- 判定条件：户名规范化后相似度高，但对手账号不一致；需结合业务背景判断是否同一交易对手。
- 处理建议：列为候选并标注"户名近似/账号不同"，要求确认对手主体后再匹配；不得仅凭户名相似自动匹配。
- 判定依据：参考方向银行结算账户户名与账号一致性核验方向（公开制度方向，非内部制度）。
- 示例：对手"XX集团" vs "XX集团财务公司"，账号不同 → 候选待确认主体关系。

本项目抽象规则：户名近似而账号不同时只能作候选，须确认对手主体一致后方可匹配，防止张冠李戴。

## [BE-R007] 金额相近含尾差的候选判定（边界：vs BE-R002/BE-R003）

- 适用场景：金额并非完全相等而是存在尾差，需区分"候选匹配后的费税差异"与"非同一笔"。
- 判定条件：金额接近但不等，尾差可由手续费/税费解释→匹配为候选并附费税说明；尾差无法解释且较大→不构成候选。
- 处理建议：尾差命中费税特征→候选+转 BE-R003 解释；尾差无法解释→不匹配，避免强行凑对。
- 判定依据：参考方向差异需保留双边口径与证据的核对方向（公开制度方向，非内部制度）。
- 示例：候选两笔差 ¥5.00 命中手续费档位 → 候选成立并标注费差；差 ¥820 无解释 → 不候选。

本项目抽象规则：候选匹配允许可被费税解释的小额尾差（联动 BE-R003），但不得对无法解释的大额差强行凑对（应回 BE-R002）。

## [BE-R007] 候选匹配确认后的台账与证据（处理规范）

- 适用场景：候选经确认后落账的证据归档，确保人工/Agent 确认过程可追溯。
- 判定条件：落账证据须含候选依据（金额/日期/对手相似度）、确认人或确认 Agent、确认时间与最终匹配关系。
- 处理建议：未记录确认主体与依据的候选不得落账；确认被否的候选回退到对应单边或差异分支。
- 判定依据：参考方向审计留痕与复核责任可追溯要求（公开制度方向，非内部制度）。
- 示例：候选确认记录 `{candidate_score, confirmed_by, ts, matched_pair}` 写入台账。

本项目抽象规则：BE-R007 候选落账必须留存"候选依据+确认主体+匹配关系"，确认被否则回退到单边/差异分支。
