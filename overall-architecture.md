# 《基于多智能体（Multi-Agent）架构的银行自动化对账与报表审计系统》总体架构

## 1. 项目定位与架构目标

本项目是一个面向个人开源作品集和转行求职展示的 AI 应用系统。项目场景来自银行对公柜员工作中常见的对账、冲正、挂账、日终报表和人工复核流程，目标是把真实业务问题抽象成一个可开发、可部署、可面试讲透的系统。

系统的核心定位不是“让大模型替人算账”，而是：

> 用确定性代码完成金融账务中的精确计算和状态落库，用 Multi-Agent 处理非结构化摘要、跨日切追溯、规则解释和审计报告生成，用 Human-in-the-Loop 保留金融场景中的人工复核边界。

架构设计需要同时服务四个目标：

1. **业务可信**：场景来自银行对账和报表审计，不是泛泛的聊天机器人。
2. **工程可落地**：先做 MVP-0 后端最小 AI 对账闭环，再逐步补本地产品闭环、线上部署和工程化能力。
3. **AI 能力明确**：突出 RAG、Agent 编排、工具调用、结构化输出和审计溯源。
4. **面试可表达**：每个技术选择都能解释“为什么这么做”，而不是堆技术名词。

### 2.1 前端交互层

前端使用 Vue 3、Element Plus、Pinia 和 ECharts，目标不是做营销页面，而是做一个可以真实演示业务流程的工作台。

核心页面包括：

- 账单上传页：上传银行端流水和清算端流水。
- 任务看板：展示总笔数、自动平账数、AI 审计中、待人工复核、挂账数。
- Agent 流式工作台：通过 SSE 展示 Agent 当前处理步骤、工具调用和 RAG 命中依据。
- 人工复核页：展示双端流水、AI 推荐理由、RAG 来源和人工操作按钮。
- 差错台账页：查询已确认差错、挂账、冲正、手续费差异等记录。
- 报表审计页：查看日终统计、异常分布和审计报告。

### 2.2 API 服务层

后端使用 FastAPI 作为统一入口，负责文件上传、任务管理、工作流启动、人工审批、台账查询和报告导出。

关键能力包括：

- Pydantic 请求和响应校验。
- JWT 登录鉴权，保护线上 Demo 接口和模型调用额度。
- SSE 流式推送 Agent 执行过程。
- 统一异常处理和审计日志。
- 环境变量管理数据库、模型 Key、向量库路径等配置。

### 2.3 确定性计算层

确定性计算层处理所有必须精确、可复现、可审计的动作。

职责包括：

- Excel 字段清洗和标准化。
- 双端流水初步匹配。
- 金额差异计算。
- 单边账识别。
- 手续费和批量业务规则初筛。
- MySQL 事务写入。

本层的核心原则是：**凡是金额计算、状态更新、数据库写入，不交给 LLM。**

### 2.4 Multi-Agent 编排层

Multi-Agent 编排层使用 LangGraph 实现状态机和节点路由。Agent 不是多个聊天机器人，而是带状态、工具、边界和失败兜底的业务节点。

建议节点如下：

- `PreCheckNode`：规则预处理节点，不依赖 LLM，完成基础对账和异常分类。
- `ExtractionAgent`：提取 Agent，将不规范摘要结构化为标准 JSON。
- `AuditAgent`：审计 Agent，结合规则、RAG 和工具结果做审计判断。
- `TraceAgent`：追溯 Agent，处理跨日切、冲正、退款和原流水追踪。
- `ReportAgent`：报表 Agent，生成审计摘要和报告。
- `HumanReviewNode`：人工复核节点，负责挂起流程并等待人工审批。

MVP-0 阶段优先实现 `PreCheckNode` 和简化 `AuditAgent`，先跑通“规则对账 -> RAG 检索 -> 审计建议 -> 台账落库”。MVP 阶段再引入 `ExtractionAgent`、`HumanReviewNode` 和 LangGraph 基础编排；V1/V2 再补齐追溯、报表和工程化增强能力。

### 2.5 数据与知识层

MySQL 8.0 存储结构化业务数据，ChromaDB 存储规则和合规知识。

MySQL 负责：

- 对账任务。
- 银行端流水。
- 清算端流水。
- 待核验队列。
- 差错台账。
- 人工复核记录。
- Agent 执行日志。
- RAG 检索记录。
- 审计报告。

ChromaDB 负责：

- 项目自定义对账规则。
- 跨日切处理规则。
- 冲正和退款识别规则。
- 手续费差异规则。
- 公开支付结算规则和报表审计口径。

### 2.6 工程化与部署层

工程化层服务于项目从本地 Demo 走向可展示作品。

MVP-0 阶段：

- 本地 FastAPI。
- Pandas 清洗和基础规则对账。
- 简化 AuditAgent。
- 本地 ChromaDB。
- MySQL 数据落库。
- 模拟 Excel 数据。

MVP 阶段：

- 本地 FastAPI。
- 本地 Vue。
- Docker MySQL。
- 本地 ChromaDB。
- 模拟 Excel 数据。
- LangGraph 基础工作流。
- 人工复核基础流程。

V1 阶段：

- Docker Compose 一键启动。
- JWT 鉴权。
- 云服务器部署。
- README、演示数据和部署说明。
- 基础日志和异常处理。

V2 阶段：

- RAG 检索评测。
- Agent 执行指标。
- Prompt 版本管理。
- 失败样本分析。
- 基础可观测性面板。

## 3. 核心设计原则

### 3.1 规则优先，AI 补充

能用确定性规则解决的问题，不交给大模型。例如金额一致、流水号一致的账目应直接自动平账；金额不一致应由 Decimal 计算差额并进入台账，而不是让 LLM 判断差多少。

面试可讲点：这体现的是 AI 应用开发中的边界意识。AI 不是替代所有逻辑，而是补足传统规则难处理的语义理解和归因解释。

### 3.2 金额计算绝不交给 LLM

金融账务中的金额计算必须可复现、可验证、可审计。系统统一使用 Python `decimal.Decimal` 和 MySQL 事务保证精确性。

Agent 只能读取工具返回的计算结果，例如：

```json
{
  "bank_amount": "1000.00",
  "clear_amount": "990.00",
  "diff": "10.00"
}
```

Agent 不允许自行做加减法，只能基于工具结果和业务规则做判断。

### 3.3 RAG 无据不判定

涉及跨日切、冲正、挂账、手续费争议等业务判断时，Agent 不能凭借预训练知识直接给结论。它必须先从 RAG 知识库检索到规则依据。

如果没有命中规则，系统强制进入 `PENDING_HUMAN`，由人工复核。

### 3.4 Human-in-the-Loop 兜底

金融场景中 AI 推荐不能等同于最终处理结果。只要出现低置信度、无规则依据、高风险金额、工具调用失败等情况，流程必须进入人工复核。

人工复核需要记录：

- 操作人。
- 操作时间。
- 操作动作。
- 人工备注。
- AI 推荐理由。
- RAG 来源。

### 3.5 全链路可追踪

系统需要记录每一次 Agent 决策过程，包括输入、工具调用、RAG 命中片段、输出 JSON、路由结果和错误信息。

这样做有两个目的：

1. 方便开发阶段调试 Agent 行为。
2. 面试时能解释 AI 判断不是黑盒，而是有日志、有依据、有兜底。

## 4. Multi-Agent 协作架构

### 4.1 全局状态结构

LangGraph 中各节点共享 `ReconciliationState`，状态结构建议如下：

```python
from typing import Any, Dict, List, Optional, TypedDict

class ReconciliationState(TypedDict):
    task_id: str
    current_queue_id: Optional[int]
    bank_item: Dict[str, Any]
    clear_item: Dict[str, Any]
    error_type: Optional[str]
    math_result: Dict[str, str]
    extraction_result: Dict[str, Any]
    rag_context: List[Dict[str, Any]]
    audit_decision: Dict[str, Any]
    retry_count: int
    next_action: str
    error_message: Optional[str]
    agent_logs: List[Dict[str, Any]]
```

### 4.2 节点职责

| 节点 | 类型 | 职责 | 阶段 |
| --- | --- | --- | --- |
| `PreCheckNode` | 确定性代码 | Excel 清洗、金额比对、异常初筛 | MVP-0 |
| `AuditAgent` | LLM Agent | 结合计算结果和 RAG 依据输出结构化审计建议 | MVP-0 |
| `ExtractionAgent` | LLM Agent | 模糊摘要结构化，输出 JSON | MVP |
| `HumanReviewNode` | 状态节点 | 挂起流程，等待人工审批 | MVP |
| `ReportAgent` | LLM + Tool | 生成审计摘要和报告 | V1/V2 |
| `TraceAgent` | LLM + Tool | 查询 T+1 流水、追溯冲正和退款链路 | V2 |

### 4.3 路由规则

MVP-0 路由：

```text
START
  -> 上传并解析双端 Excel
  -> PreCheckNode 完成字段校验、数据清洗、基础对账和异常识别
  -> 异常交易进入 RAG 检索
  -> AuditAgent 输出结构化审计建议
  -> 写入 MySQL 差错台账和任务统计
  -> END
```

MVP 及后续路由：

```text
START
  -> PreCheckNode
  -> 基础差错直接写入台账
  -> 模糊摘要进入 ExtractionAgent
  -> 日切/冲正/规则争议进入 RAG 检索
  -> AuditAgent 给出结构化判断
  -> 低置信度或无依据进入 HumanReviewNode
  -> 处理完成进入 ReportAgent 或 END
```

### 4.4 失败处理

- JSON 解析失败：最多重试 3 次。
- RAG 未命中：转人工复核。
- 工具调用失败：记录日志并转人工复核。
- 数据库事务失败：回滚并标记任务失败。
- Agent 输出不符合 Schema：拒绝落库并重试。

## 5. RAG 合规知识库与业务规则中心

RAG 是本项目的核心模块，不只是给 Agent 补充上下文。它承担“规则依据、业务解释、审计溯源”三类职责。

### 5.1 数据来源与证据流

本项目的数据和审计依据采用分层设计：

```text
人工构造模拟 Excel
  -> Pandas 清洗和规则对账
  -> 异常交易上下文
  -> RAG 检索公开依据摘要和项目规则
  -> AuditAgent 生成结构化审计建议
  -> MySQL 保存差错台账、RAG 来源和审计结果
```

数据来源只使用人工构造的模拟数据。银行端流水和清算端流水的字段结构参考对账业务需要设计，但客户姓名、账号、流水号、金额和摘要均为虚构或脱敏样式。

审计依据由三层组成：

- **公开制度依据**：中国人民银行公开支付结算制度、银行结算账户管理制度，以及财政部公开会计基础工作、企业内部控制和会计档案管理规范。
- **项目自定义业务规则**：面向演示场景编写的 Markdown 规则，例如基础对账、跨日切、冲正退款、人工复核和报表口径。
- **运行证据**：RAG 命中的规则来源、相似度分数、AuditAgent 输出 JSON、人工复核记录和差错台账记录。

公开制度依据只用于说明项目遵循规则、凭证、内控和留痕原则；项目不使用银行内部制度，也不把演示规则包装成真实银行制度。

### 5.2 RAG 解决的问题

1. **防止幻觉**：Agent 没有检索到规则时不能自动判定。
2. **规则可维护**：业务规则写成 Markdown 文档，而不是全部塞进 Prompt。
3. **审计可追溯**：每个关键判断保存引用来源。
4. **对齐 JD 技能**：覆盖文档处理、切片、Embedding、向量检索、阈值、Rerank、评测。

### 5.3 知识库内容

MVP-0 阶段使用项目自定义规则文档：

- 基础对账规则。
- 跨日切单边账处理规则。
- 冲正和退款识别规则。
- 人工复核触发规则。

MVP 阶段补充：

- 手续费差异处理规则。
- 报表审计口径说明。

V1/V2 阶段可加入公开规则和 PDF/Word 文档解析，但不使用任何银行内部资料或客户数据。

### 5.4 RAG 流程

```text
规则文档 Markdown/PDF
  -> 文档清洗
  -> 按标题、条款、业务规则切片
  -> 中文 Embedding 向量化
  -> 存入 ChromaDB
  -> 根据当前账务上下文检索 Top-K
  -> 相似度阈值过滤
  -> 可选 Rerank
  -> 返回 rag_context 给 AuditAgent
  -> 保存 rag_source 和检索分数
```

### 5.5 检索策略

MVP-0：

- Markdown 文档。
- ChromaDB。
- Top-K 检索。
- 简单相似度阈值。
- RAG 检索调试接口。
- AuditAgent 输出 evidence 字段。

MVP：

- 中文 Embedding 模型。
- 在前端展示引用来源。

V1：

- PDF/Word 解析。
- 公开规则文档导入。

V2：

- Rerank。
- 检索评测集。
- 命中率统计。
- 失败样本分析。

### 5.6 无命中策略

如果 RAG 没有命中可用规则，`AuditAgent` 必须输出：

```json
{
  "decision": "PENDING_HUMAN",
  "reason": "未检索到足够的业务规则或合规依据，转人工复核",
  "rag_source": []
}
```

## 6. 数据流与状态流转

### 6.1 主流程

MVP-0 主流程：

1. 准备模拟银行端流水和清算端流水。
2. 通过 FastAPI 上传两份 Excel。
3. Pandas 读取文件，完成字段校验、清洗和标准化。
4. 基础规则对账，识别自动平账数据和异常交易。
5. 异常交易进入 RAG 检索，获取规则依据。
6. AuditAgent 输出结构化审计建议。
7. 结果写入 MySQL 差错台账和任务统计。
8. 通过 API 查询任务状态和差错明细。

MVP 及后续主流程：

1. 用户上传银行端流水和清算端流水。
2. FastAPI 保存文件并创建对账任务。
3. Pandas 清洗字段，完成基础匹配。
4. 完全匹配数据直接计入自动平账统计。
5. 异常数据写入待核验队列。
6. LangGraph 启动 Agent 工作流。
7. Agent 根据异常类型调用 RAG 和业务工具。
8. 可自动处理的结果写入差错台账或标记已平账。
9. 不确定结果进入人工复核。
10. 任务完成后生成审计报告。

### 6.2 状态定义

| 状态 | 含义 |
| --- | --- |
| `UPLOADED` | 文件已上传 |
| `PRECHECKING` | 预处理中 |
| `PENDING_AI` | 待 AI 审计 |
| `AI_RUNNING` | AI 审计中 |
| `PENDING_HUMAN` | 待人工复核 |
| `FIXED` | 已平账 |
| `UNRESOLVED` | 挂账或未解决 |
| `FAILED` | 系统处理失败 |
| `REPORTED` | 已生成报告 |

## 7. 数据与安全边界

本项目是个人开源学习项目，必须明确安全边界：

- 只使用模拟数据和脱敏数据。
- 模拟数据由项目人工构造，用于覆盖正常平账、金额差错、单边缺失、跨日切、冲正退款、手续费差异、重复入账和漏记账等场景。
- 不使用任何真实客户数据。
- 不使用任何银行内部资料。
- 公开制度依据只作为项目规则设计参考，不等同于真实银行内部审计制度。
- 项目自定义规则必须标注为演示规则。
- 不宣称系统可直接用于真实生产银行系统。
- AI 只做辅助分析和建议，不做最终金融决策。
- 金额计算、状态落库、人工审批由确定性代码和数据库事务保障。

## 10. 面试可讲点

这个架构可以重点讲五件事：

1. **业务理解**：项目不是抽象 AI Demo，而是从银行对账、冲正、挂账和报表审计中抽象出来的。
2. **AI 边界**：金额计算、状态落库、审批结果不交给 LLM。
3. **RAG 价值**：Agent 的审计判断必须有规则来源，没有依据就转人工。
4. **后端能力**：FastAPI、MySQL 事务、状态机、SSE、JWT、Docker 都能在项目里落地。
5. **学习路径**：MVP-0、MVP、V1、V2 的拆分体现了个人项目的可执行性；先验证后端最小 AI 对账闭环，再做本地产品、在线作品和简历最终版。
