# 《基于多智能体（Multi-Agent）架构的银行自动化对账与报表审计系统》需求梳理

## 1. 需求来源与项目定位

本项目是一个个人开源求职项目，目标岗位指向 AI Agent 开发工程师、AI 应用开发工程师和 Python 后端开发工程师。项目选题来自银行对公柜员经历：日常业务中会接触转账、回单、冲正、手续费、挂账、日终轧账和人工复核等流程，这些流程天然适合作为“真实业务 + AI 工程化”的项目载体。

项目要解决的问题不是简单做一个 Excel 对账脚本，而是构建一个完整的对账与审计工作流：

- 传统代码负责精确计算、规则初筛和数据库事务。
- AI Agent 负责非结构化摘要理解、跨日切追溯、异常归因和报告生成。
- RAG 负责提供规则依据，避免大模型凭空判断。
- 人工复核负责兜底金融风险，保留可审计的处理记录。

这个项目的核心目标是让面试官看到：候选人既理解银行业务，也能把业务问题拆成可开发的软件系统，并逐步掌握主流 AI 应用开发技术栈。

## 2. 用户画像

### 2.1 银行运营或对账人员

关注点：

- 今日账是否平。
- 哪些流水异常。
- 哪些异常可以自动处理。
- 哪些需要人工复核。
- 最终台账和报告是否可追溯。

### 2.2 柜员或业务复核人员

关注点：

- 手工冲正、退款、挂账原因是否清楚。
- AI 推荐勾平是否有依据。
- 复核操作是否简单。
- 操作后是否留下记录。

### 2.3 技术面试官

关注点：

- 项目是否来自真实业务理解。
- 是否只是套模板或调用大模型 API。
- 后端、数据库、RAG、Agent、部署能力是否有实际设计。
- 候选人是否知道 AI 在金融场景里的边界。

## 3. 核心业务痛点

### 3.1 对账不是简单金额相等

真实对账中，金额一致只是最简单情况。系统还需要处理单边账、跨日切、冲正、手续费、批量业务失败、重复入账、漏记账等问题。

### 3.2 柜员摘要存在非结构化噪声

手工摘要常常不规范，例如“冲红昨日错账”“反向抹账”“客户错汇退回”。传统关键词规则容易漏判或误判，需要 AI 做语义结构化。

### 3.3 跨系统日切导致短期单边

银行核心、清算系统和第三方支付系统的日切时间可能不同，导致当天看似缺账，次日才能补齐。系统需要识别时间窗口并追溯 T+1 流水。

### 3.4 合规和审计依据需要可追溯

金融场景不能让大模型直接说“可以平账”。关键判断必须有规则依据；没有依据时必须进入人工复核。

### 3.5 报表整理耗时且容易遗漏

对账结束后，还需要汇总差错类型、处理状态、人工复核数量、挂账金额和异常原因。这个过程适合由 ReportAgent 辅助生成审计报告。

## 4. 数据来源与审计依据

### 4.1 数据来源

本项目不使用真实银行流水、真实客户信息或银行内部资料。演示数据全部由项目人工构造，字段和异常模式来自银行对账业务抽象。

数据分为两类：

- 银行端流水：模拟银行核心、网银或柜面系统中的入账/出账流水。
- 清算端流水：模拟清算平台、第三方支付渠道或批量业务系统中的清算流水。

模拟数据需要覆盖：

- 正常平账。
- 基础金额差错。
- 单边缺失。
- 跨日切 T+1 补记。
- 冲正、退款、反向抹账。
- 手续费和批量业务差异。
- 重复入账、漏记账和疑似重复清算。

后续工程中建议为模拟数据单独维护说明文档，写清楚字段含义、样本场景、预期识别结果和“数据全部虚构”的边界。

### 4.2 审计依据来源

审计依据由三层组成。

第一层是公开制度依据，用于说明项目遵循支付结算、会计基础、内部控制和资料留痕等通用原则。可参考：

- 中国人民银行《支付结算办法》。
- 中国人民银行《人民币银行结算账户管理办法》。
- 财政部《会计基础工作规范》。
- 财政部等五部委《企业内部控制基本规范》。
- 财政部、国家档案局《会计档案管理办法》。

第二层是项目自定义业务规则，用 Markdown 文档维护，例如基础对账规则、跨日切规则、冲正退款规则、人工复核规则、手续费规则和报表口径规则。这些规则是演示用业务规则，不冒充银行内部制度。

第三层是系统运行证据，包括 RAG 命中的规则来源、相似度分数、AuditAgent 结构化输出、人工复核记录和差错台账记录。

### 4.3 Agent 审计边界

AuditAgent 的输出必须包含依据。没有 RAG 命中、命中分数不足、工具调用失败或结构化输出不完整时，Agent 不允许自动平账，只能给出待人工复核或待处理建议。

核心原则：

- 数据可以模拟，但来源和生成逻辑要说清楚。
- 规则可以自定义，但必须标注为项目演示规则。
- 公开制度可以作为原则依据，但不能包装成银行内部制度。
- AI 可以解释和建议，但不能替代金额计算、状态落库和最终复核。

## 5. 六类核心业务场景

### 5.1 场景一：基础金额差错

#### 业务描述

双端流水号匹配，但金额不一致；或者一端存在流水，另一端缺失。常见表现包括系统短款、长款、漏记账和单边缺失。

#### 系统处理

- Pandas 完成字段清洗和初步匹配。
- `decimal.Decimal` 计算差额。
- MySQL 事务写入差错台账。
- MVP-0 阶段通过 API 查询差错明细；MVP 阶段前端在差错台账中标注差异金额。

#### AI 参与边界

该场景不需要 LLM 计算金额。AI 最多用于后续解释差错类型，不能参与金额计算。

#### 面试可讲点

这个场景体现“规则优先，AI 补充”的原则。金融项目里并不是所有问题都要交给大模型，精确计算必须交给确定性代码。

### 5.2 场景二：跨日切单边账

#### 业务描述

交易发生在深夜日切敏感窗口，一端当天有账，另一端次日才出现对应流水。传统系统可能直接判定为单边差错。

#### 系统处理

- PreCheckNode 识别交易时间是否落在日切窗口。
- RAG 检索跨日切处理规则。
- MVP 阶段通过基础工具查询 T+1 流水；V2 阶段由 TraceAgent 增强追溯链路。
- 找到对应流水后自动标记为可平账。
- 未找到或无规则依据时转人工复核。

#### AI 参与边界

Agent 负责判断是否需要追溯和调用查询工具，但最终金额匹配仍由确定性代码完成。

#### 面试可讲点

这个场景可以说明 Agent 不是聊天机器人，而是能基于业务状态调用工具完成追溯任务。

### 5.3 场景三：冲正、退款、反向抹账

#### 业务描述

柜员或系统产生冲正、退款、反向抹账等交易，但摘要不统一，传统规则难以稳定识别。

示例：

- “手工调账-冲红昨日错账”
- “客户错汇退回”
- “网联反向抹账”
- “退款冲正原交易”

#### 系统处理

- ExtractionAgent 将摘要结构化为标准 JSON。
- AuditAgent 判断是否构成冲正链路。
- RAG 检索冲正和退款处理规则。
- 低置信度结果进入 Human-in-the-Loop。

#### AI 参与边界

AI 负责语义理解和结构化，不负责直接修改账务状态。状态更新必须由后端接口和数据库事务执行。

#### 面试可讲点

这个场景体现 Prompt Engineering、结构化输出、JSON Schema、RAG 引用和人工复核。

### 5.4 场景四：手续费、渠道费、批量业务差异

#### 业务描述

对公转账、批量代发、批量代扣和第三方清算业务中，可能出现手续费、渠道费或部分失败导致的金额差异。

#### 系统处理

- 规则引擎优先判断已知手续费规则。
- 无法直接解释时调用 RAG 检索业务规则。
- Agent 给出差异归因建议。
- 高风险或规则不充分时进入人工复核。

#### AI 参与边界

AI 不计算手续费，只解释规则和差异原因。手续费计算由规则引擎或确定性工具完成。

#### 面试可讲点

这个场景把项目从简单对账扩展到更接近银行真实业务，也能体现规则引擎和 Agent 的协作。

### 5.5 场景五：重复入账、漏记账、疑似重复清算

#### 业务描述

同一客户、同一金额、相近时间出现多笔相似流水，可能是重复入账、重复清算或漏记账导致的异常。

#### 系统处理

- 使用客户、金额、时间窗口、摘要相似度进行候选异常识别。
- Agent 对异常进行归因解释。
- 涉及资金风险的结果转人工复核。

#### AI 参与边界

AI 可以解释相似流水之间的可能关系，但不能直接做最终核销。

#### 面试可讲点

这个场景体现数据处理、相似匹配、异常检测和 Agent 解释能力，适合作为 V2 加分项。

### 5.6 场景六：日终报表审计

#### 业务描述

对账完成后，需要生成日终审计报告，说明本批次总笔数、总金额、自动平账率、挂账率、人工复核数量和异常类型分布。

#### 系统处理

- ReportAgent 读取 MySQL 汇总数据。
- RAG 检索报表口径和异常解释规则。
- 生成 Markdown 审计报告。
- V2 支持 PDF 导出。

#### AI 参与边界

统计指标由 SQL 和确定性代码生成，AI 只负责把结构化结果解释成审计摘要。

#### 面试可讲点

这个场景让项目形成“对账 -> 复核 -> 台账 -> 报告”的完整业务闭环。

## 6. 功能模块拆解

### F1. 账单上传与预处理

阶段：MVP-0 起。

后端：

- 接收双端 Excel。
- 校验文件格式和字段。
- 清洗金额、时间、流水号、摘要。
- 自动过滤完全匹配数据。
- 写入对账任务和待核验队列。

前端：

- MVP-0 不建设上传页面，通过 API 上传。
- 拖拽上传。
- 字段校验提示。
- 上传结果统计。

### F2. AI 对账工作台

阶段：MVP 起，SSE 流式展示在 V1 完成。

后端：

- MVP-0 先执行简化 AuditAgent 审计流程。
- MVP 启动 LangGraph 基础工作流。
- V1 通过 SSE 推送 Agent 执行事件。
- MVP 起记录 Agent 日志。

前端：

- 展示当前处理流水。
- 展示工具调用。
- 展示 RAG 命中规则。
- 展示 AI 决策结果。

### F3. RAG 合规知识库与规则中心

阶段：MVP-0 起。

后端：

- 导入 Markdown 规则文档。
- 文档切片和向量化。
- ChromaDB 检索。
- 保存检索记录。

前端：

- MVP-0 和 MVP 可不做管理页。
- V1/V2 支持规则文档查看和检索调试。

### F4. 人工复核中心

阶段：MVP 起。MVP-0 中 AI 无法判断的结果先写入任务状态和差错台账，不建设人工复核页面。

后端：

- 查询待人工复核记录。
- 接收人工审批结果。
- 使用事务更新队列、台账和复核记录。

前端：

- 双栏展示银行端和清算端流水。
- 展示 AI 推荐理由和 RAG 来源。
- 提供确认平账、强制挂账、人工备注等操作。

### F5. 差错台账

阶段：MVP-0 起。

后端：

- 提供分页查询。
- 支持按任务、状态、差错类型、日期筛选。
- 保存处理人和处理时间。

前端：

- MVP-0 不建设台账页面，通过 API 查询。
- 表格展示。
- 条件筛选。
- 异常详情查看。

### F6. 报表审计

阶段：V1 起，V2 增强。

后端：

- 聚合任务统计数据。
- 调用 ReportAgent 生成审计摘要。
- 保存报告结果。

前端：

- 展示统计指标。
- 展示异常类型分布图。
- 支持 Markdown 报告查看。

### F7. 系统管理与安全

阶段：V1 起。

后端：

- JWT 登录鉴权。
- 环境变量配置。
- 接口访问控制。
- 日志记录。

前端：

- 登录页。
- 当前用户展示。
- 登出。

## 7. 分阶段需求范围

本项目采用 MVP-0 -> MVP -> V1 -> V2 的递进路线。MVP-0 是 MVP 的前置子集，用于先验证核心业务链路和 AI 审计链路；V2 是最终写入简历并支撑面试深挖的版本。

### 7.1 MVP-0：后端最小 AI 对账闭环

MVP-0 目标是先把后端主链路跑通。

必须包含：

- 准备模拟银行端流水和清算端流水 Excel。
- 维护模拟数据说明，标注字段含义、样本场景和预期结果。
- 通过 FastAPI 上传两份模拟 Excel。
- Pandas 字段校验、数据清洗和标准化。
- 基础规则对账。
- 基础金额差错和单边缺失识别。
- MySQL 任务、流水和差错台账落库。
- MVP-0 RAG：Markdown 规则 + ChromaDB + Top-K 检索。
- 简化 AuditAgent 输出结构化审计建议。
- AuditAgent 输出包含 evidence 字段，可追溯到规则来源。
- API 查询任务状态和差错明细。

暂不包含：

- Vue 前端页面。
- 登录鉴权。
- SSE Agent 执行日志。
- 人工复核页面。
- Markdown/PDF 报告。
- 多 Agent 完整编排。
- 云服务器部署。

### 7.2 MVP：本地可演示产品闭环

MVP 目标是在 MVP-0 基础上补齐本地可演示的产品界面和人工复核流程。

新增：

- Vue 账单上传页。
- 任务看板。
- 差错台账页。
- 人工复核基础页。
- LangGraph 基础工作流。
- ExtractionAgent、AuditAgent 和 HumanReviewNode。
- 跨日切单边账样例。
- 模糊冲正样例。
- Agent 执行日志入库。

暂不包含：

- 云服务器部署。
- 复杂权限系统。
- SSE 流式工作台。
- RAG Rerank。
- PDF 报告。
- 完整监控。

### 7.3 V1：在线作品版

V1 目标是让项目能作为 GitHub + 在线 Demo 展示。

新增：

- Docker Compose。
- JWT 登录。
- 云服务器部署。
- SSE Agent 执行日志。
- 手续费/批量业务差异。
- Markdown 审计报告。
- README、演示数据和部署说明。

### 7.4 V2：简历最终版

V2 目标是补充 AI 工程深度，形成最终出现在简历上的版本。

新增：

- TraceAgent 增强。
- ReportAgent 增强。
- 重复入账和漏记账异常识别。
- 异常归因增强。
- RAG Rerank。
- RAG 检索评测。
- Prompt 版本管理。
- Agent 执行日志分析。
- PDF 报告导出。
- 基础可观测性指标。
- 失败样本分析。

## 8. 非功能性需求

### 8.1 准确性

- 金额计算必须使用 Decimal。
- 数据库状态更新必须使用事务。
- AI 输出必须经过 JSON Schema 或 Pydantic 校验。

### 8.2 安全性

- 只使用模拟数据和脱敏数据。
- 不接入真实客户数据。
- 不使用银行内部资料。
- 公开制度依据只作为项目规则设计参考。
- 项目自定义规则必须明确为演示规则。
- 线上接口使用 JWT 保护。
- 大模型 API Key 使用环境变量管理。

### 8.3 可追溯性

- MVP-0 保存 AuditAgent 结构化审计建议和 RAG 来源；MVP 起保存更完整的 Agent 日志。
- 每次 RAG 检索保存来源。
- 每次人工审批保存操作人、时间和备注。
- 模拟数据样本保存场景说明和预期结果。

### 8.4 性能与成本

- 绝大多数正常账目由 Pandas 和规则层过滤。
- 只把疑难异常交给 LLM。
- RAG 文档本地向量化，降低外部依赖。

### 8.5 可维护性

- 业务规则文档化。
- API 使用 Pydantic 约束。
- 数据库表按职责拆分。
- Agent 节点职责清晰。

## 10. 面试可讲点

1. **为什么做这个项目**：转行回计算机开发，需要一个和个人经历强相关、又能体现 AI 应用能力的项目。
2. **为什么选银行对账**：对账、冲正、挂账、日终报表是银行业务中真实存在的问题，适合做业务系统。
3. **为什么不是纯 AI**：金融场景里金额计算和状态落库必须确定性，AI 只处理语义和解释。
4. **RAG 的价值是什么**：让 Agent 的审计判断有规则来源，避免凭空编造。
5. **数据和依据从哪里来**：数据全部模拟，依据来自公开制度原则、项目自定义规则和 RAG 运行证据，不使用真实客户数据或银行内部资料。
6. **如何分阶段实现**：先做 MVP-0 后端最小 AI 对账闭环，再做 MVP 本地可演示产品，V1 上线展示，V2 补评测、报告、可观测性和失败样本分析。
7. **如何体现开发能力**：项目覆盖后端接口、数据库事务、前端工作台、Agent 编排、RAG、部署和测试。
