# 《基于多智能体（Multi-Agent）架构的银行自动化对账与报表审计系统》文档优化设计

日期：2026-05-21  
目标文档：`总体架构.md`、`需求梳理.md`、`系统PRD.md`  
参考文档：`个人背景.md`、`任职要求.md`  
项目定位：个人开源求职项目，用于展示银行业务理解、后端开发能力、AI Agent 应用工程能力与分阶段学习交付能力。

## 1. 优化目标

本次优化不是简单扩写现有文档，而是把项目从“一个金融对账 Agent Demo”升级为“一个有真实业务来源、可分阶段实现、可面试讲透的 AI 应用系统设计”。

优化后的文档需要同时服务四个目标：

1. 体现个人银行业务背景：项目场景来自对公柜员经历，业务不是凭空包装。
2. 对齐 AI Agent / AI 应用开发 JD：覆盖 FastAPI、MySQL、RAG、LangGraph、Prompt、工具调用、Human-in-the-Loop、部署、评测等能力点。
3. 降低转行学习风险：系统按 MVP、V1、V2 分阶段实现，先完成闭环，再逐步补工程深度。
4. 方便面试表达：每个关键设计都有“设计动机”和“面试可讲点”，让面试官能看到这是候选人自己构建出来的项目。

项目的核心表达应当是：

> 这个项目不是为了证明我会调用大模型 API，而是为了证明我理解银行真实业务流程，也能把业务问题抽象成可落地的软件系统，并用 AI Agent、RAG、后端工程和前端交互把它完整实现出来。

## 2. 总体优化路线

采用“分层融合型”方案：

- 业务主线：银行自动化对账与报表审计。
- AI 主线：Multi-Agent、RAG、Tool Calling、结构化输出、Human-in-the-Loop。
- 工程主线：FastAPI、MySQL、Vue、Docker、JWT、日志、测试、部署。
- 成长主线：MVP 本地闭环 -> V1 在线作品 -> V2 面试加分增强。

该路线避免两个极端：

- 只写银行业务，导致项目像普通后台管理系统。
- 只堆 Agent 概念，导致项目脱离真实业务，看起来像套模板。

## 4. 业务场景体系优化

现有文档主要覆盖金额不匹配、跨日切、模糊摘要三个场景。优化后扩展为六类场景，并按阶段实现。

### 4.1 场景一：基础金额差错

业务内容：

- 金额不一致。
- 单边缺失。
- 流水号匹配但金额不符。

技术处理：

- 不调用 LLM。
- 使用 Pandas 做初筛。
- 使用 `decimal.Decimal` 做金额计算。
- 使用 MySQL 事务写入差错台账。

设计动机：

金融账务计算必须可验证、可复现，不能依赖大模型。

面试可讲点：

体现“规则优先，AI 补充”的工程判断。

### 4.2 场景二：跨日切单边账

业务内容：

- 银行核心、清算系统、第三方支付平台日切时间不同，导致当天单边、次日补齐。

技术处理：

- Agent 识别交易时间是否落在日切敏感窗口。
- RAG 检索日切处理规则。
- TraceAgent 调用工具查询 T+1 流水。
- 命中后自动勾平，未命中转人工复核。

设计动机：

跨日切不是简单缺账，它需要结合业务规则和跨期追溯。

面试可讲点：

Agent 不只是聊天，而是通过工具调用完成业务追溯。

### 4.3 场景三：冲正、退款、反向抹账

业务内容：

- 手工摘要不规范，例如“冲红昨日错账”“反向抹账”“客户错汇退回”。

技术处理：

- ExtractionAgent 将非结构化摘要转成标准 JSON。
- AuditAgent 判断是否构成可勾平链路。
- 置信度不足或缺少依据时进入 Human-in-the-Loop。

设计动机：

柜员手工摘要具有真实业务噪声，适合展示大模型语义理解能力。

面试可讲点：

体现 Prompt、结构化输出、JSON Schema、人工复核和审计留痕能力。

### 4.4 场景四：手续费、渠道费、批量业务差异

业务内容：

- 对公转账手续费差异。
- 批量代发、代扣中的部分失败。
- 清算渠道费用和入账金额差异。

技术处理：

- 规则引擎优先识别已知手续费规则。
- RAG 提供规则解释。
- Agent 对无法解释的差异做归因建议。

设计动机：

对公业务不仅是单笔转账，还存在手续费、批量业务和渠道差异。

面试可讲点：

体现业务复杂度、规则引擎和 AI 协作。

### 4.5 场景五：重复入账、漏记账、疑似重复清算

业务内容：

- 同客户、同金额、相近时间出现多笔相似流水。
- 一端存在重复清算或漏记账。

技术处理：

- 使用规则和相似度检索发现候选异常。
- Agent 做异常归因解释。
- 高风险结果进入人工复核。

设计动机：

部分异常不是单纯金额差，而是需要从时间、金额、客户、摘要等多维度识别。

面试可讲点：

体现数据处理、异常检测、相似匹配和 Agent 解释能力。

### 4.6 场景六：日终报表审计

业务内容：

- 对账完成后生成日终报表。
- 审计报表中的总笔数、总金额、平账率、挂账率、人工复核数、异常类型分布。

技术处理：

- ReportAgent 读取 MySQL 汇总数据。
- RAG 提供审计口径和解释规则。
- 输出 Markdown/PDF 审计报告。

设计动机：

把系统从“对账工具”升级为“对账 + 审计 + 报表”的完整业务闭环。

面试可讲点：

体现 AI 应用落地、报表生成、数据分析和业务闭环能力。

## 7. RAG 模块专项设计

RAG 应从原文档中的辅助能力升级为核心模块，命名为“RAG 合规知识库与业务规则中心”。

### 7.1 RAG 的系统职责

RAG 在本项目中承担三类职责：

1. 合规依据检索：为跨日切、挂账、冲正、手续费争议等提供规则来源。
2. 业务规则解释：把项目规则文档化后供 Agent 检索，而不是全部写死在 Prompt 中。
3. 审计报告溯源：报告中的关键判断需要带规则来源和检索依据。

### 7.2 文档来源

MVP 阶段：

- 项目自定义 Markdown 规则文档。
- 对账处理规则。
- 跨日切处理规则。
- 冲正和退款识别规则。
- 人工复核触发规则。

V1/V2 阶段：

- 公开监管规则。
- 支付结算通用规范。
- 报表审计口径说明。
- PDF / Word 文档解析。

### 7.3 处理流程

RAG 流程：

1. 文档导入。
2. 文档清洗。
3. 按标题、条款、业务规则切片。
4. 使用中文 Embedding 模型向量化。
5. 存入 ChromaDB。
6. 根据当前账务上下文检索 Top-K。
7. 结合相似度阈值过滤低质量结果。
8. 将命中的规则片段交给 AuditAgent。
9. 保存 `rag_source`、`rag_context` 和检索分数。

### 7.4 阈值与兜底策略

原则：

- 命中规则且分数达标：允许 Agent 结合规则做判断。
- 命中规则但分数较低：进入人工复核。
- 没有命中规则：强制转人工复核。

这样可以避免 Agent 在金融场景中凭空编造依据。

### 7.5 学习路径

RAG 学习应分三步：

1. MVP RAG：Markdown 规则文档 + ChromaDB + Top-K 检索。
2. 增强 RAG：PDF 解析、中文 Embedding、相似度阈值、来源引用。
3. 加分 RAG：Rerank、检索评测集、命中率统计、失败样本分析。

## 9. Agent 工作流优化方向

### 9.1 状态定义

建议统一状态：

- `UPLOADED`：文件已上传。
- `PRECHECKING`：预处理中。
- `PENDING_AI`：待 AI 审计。
- `AI_RUNNING`：AI 审计中。
- `PENDING_HUMAN`：待人工复核。
- `FIXED`：已平账。
- `UNRESOLVED`：挂账/未解决。
- `FAILED`：系统处理失败。
- `REPORTED`：已生成报告。

### 9.2 路由原则

- 基础金额差错：确定性计算层直接处理。
- 日切类异常：进入 TraceAgent + RAG。
- 模糊摘要类异常：进入 ExtractionAgent + AuditAgent。
- 无依据或高风险：进入 HumanReviewNode。
- 完成后统一进入台账和报告链路。

### 9.3 错误处理

- JSON 解析失败：最多重试 3 次。
- RAG 无命中：不允许自动判定，转人工。
- 工具调用失败：记录 Agent 执行日志，转人工或标记失败。
- 数据库事务失败：回滚并记录错误。
