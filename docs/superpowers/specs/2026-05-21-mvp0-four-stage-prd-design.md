# 银行对账 Agent 系统 PRD 四阶段调整设计

日期：2026-05-21  
目标文档：`系统PRD.md`、`总体架构.md`、`需求梳理.md`  
调整目标：将原有 MVP -> V1 -> V2 三阶段，重构为 MVP-0 -> MVP -> V1 -> V2 四阶段。

## 1. 设计结论

采用“主链路递进型四阶段”。

四个版本不是互相独立的功能包，而是围绕同一条核心业务链路逐层加厚：

```text
MVP-0：验证主链路能跑通
MVP：把主链路做成可本地演示的产品闭环
V1：把本地产品变成可部署、可展示的在线作品
V2：把在线作品增强成最终写进简历的 AI Agent 工程项目
```

MVP-0 是 MVP 的前置子集，不作为最终产品版本，而是用于降低开发风险、验证核心业务链路和 AI 审计链路是否成立。

## 2. MVP-0 边界

MVP-0 定位为“后端最小 AI 对账闭环”。

核心链路：

```text
模拟 Excel 数据
-> 上传银行端流水 + 清算端流水
-> Pandas 读取、字段校验、数据清洗
-> 基础规则对账
-> 识别异常交易
-> 异常进入 Agent 审计流程
-> RAG 检索规则依据
-> AuditAgent 输出结构化审计建议
-> 结果写入 MySQL 差错台账
-> API 查询任务状态和差错明细
```

MVP-0 包含：

- 模拟 Excel 数据。
- FastAPI 上传接口。
- Pandas 清洗与字段校验。
- 基础规则对账。
- 异常识别。
- 简化 AuditAgent。
- Markdown 规则文档 + ChromaDB Top-K 检索。
- AuditAgent 结构化 JSON 输出。
- MySQL 差错台账。
- API 查询任务状态和差错明细。

MVP-0 不包含：

- Vue 前端。
- 登录鉴权。
- SSE 流式事件。
- 人工复核页面。
- 报告生成。
- 多 Agent 完整编排。
- Docker Compose 一键部署。

## 3. 后续阶段边界

目标：把 MVP-0 从“后端链路”变成“本地可演示系统”。

- 差错台账页。
- 人工复核基础页。

## 4. 文档修改点

新增“数据来源与审计依据设计”：

- 数据来源：只使用人工构造的模拟 Excel 数据，字段和异常模式来自银行对账业务抽象，不使用真实客户数据、真实流水或银行内部资料。
- 数据样本：覆盖正常平账、金额差错、单边缺失、跨日切、冲正退款、手续费差异、重复入账和漏记账等场景。
- 审计依据：由公开制度依据、项目自定义业务规则和 Agent/RAG 检索记录三层组成。
- 公开制度依据：可引用中国人民银行公开支付结算制度、银行结算账户管理制度，以及财政部公开会计基础工作、企业内部控制和会计档案管理规范。
- 项目自定义业务规则：以 Markdown 形式维护，用于演示对账处理、跨日切、冲正退款、人工复核和报表口径。
- Agent 约束：AuditAgent 必须引用 RAG 命中的规则来源；没有依据或置信度不足时，不允许自动判定，必须进入人工复核或待处理状态。

- 第 1 章文档信息：项目阶段改为 MVP-0 -> MVP -> V1 -> V2。

## 5. 自查结果

- 无占位符或待确认项。
- 四阶段关系一致：MVP-0 是 MVP 子集，V2 是简历最终版。
- MVP-0 范围足够小，但保留 Agent/RAG，不会让 AI 能力像后期硬接。
- 修改范围聚焦在阶段边界，不改变项目总体业务定位。
